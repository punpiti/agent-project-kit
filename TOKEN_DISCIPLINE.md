# TOKEN_DISCIPLINE — คุณภาพสูง แต่ใช้ token ให้ฉลาดขึ้น

ผู้ใช้ให้ความสำคัญกับคุณภาพงานมากกว่า token saving แบบสุดโต่ง แต่ต้องการให้ AI ช่วยเตือนเมื่อกำลังใช้ token เปลืองโดยไม่จำเป็น

## Default Principle

> อย่าประหยัดจนงานแย่ แต่อย่าอ่าน/อธิบาย/วนซ้ำเกินจำเป็น

## Token Modes

| Mode | ใช้เมื่อ | พฤติกรรมที่ควรทำ |
|---|---|---|
| `T0 Quick` | งานเล็ก, ขอ wording, ขอ command, ขอเช็กจุดเดียว | ตอบสั้น ตรวจเฉพาะจุด ไม่สแกนทั้ง project |
| `T1 Standard` | งานทั่วไป | อ่านไฟล์ที่เกี่ยวข้อง ทำ spec/eval แบบย่อ สรุป risk |
| `T2 Deep` | งานวิจัย, reviewer, architecture, bug ยาก, policy เสี่ยง | วิเคราะห์ลึก แต่ต้องมี scope และ review gate ชัด |
| `T3 Agentic Run` | ให้ Codex แก้/ทดสอบ/วนหลายรอบ | ทำเป็นรอบเล็ก จำกัด loop ไม่ให้แก้มั่ว รายงาน test และ blocker |

ค่าเริ่มต้นคือ `T1 Standard` เว้นแต่งานชัดว่าเล็กหรือหนัก

## Cost Profiles

Token mode คือความลึกของงาน ส่วน cost profile คือความไวต่อค่าใช้จ่ายของผู้ใช้
หรือข้อจำกัดของ AI client/model ที่ใช้อยู่

| Profile | ใช้เมื่อ | พฤติกรรมที่ควรทำ |
|---|---|---|
| `C0 Economy` | ผู้ใช้บอกว่าต้องประหยัด, ใช้โมเดลตัวถูก/จำกัด, หรือ task อาจกิน token หนัก | ถามก่อน deep scan / web search / test loop ยาว ๆ และเสนอทางเลือกถูกกว่า |
| `C1 Normal` | ค่าเริ่มต้นเมื่อไม่รู้ข้อจำกัด | ทำงานตาม T1 Standard แต่เตือนก่อนขยาย scope ที่กิน token ชัดเจน |
| `C2 Premium` | ผู้ใช้บอกว่าคุณภาพ/ความครบสำคัญกว่า token หรือใช้โมเดล/แผนที่รับงานหนักได้ | ไม่ต้องถามจุกจิกก่อนอ่าน/ทดสอบที่จำเป็น แต่ยังต้องไม่อ่านเกิน scope |

ถ้าเดา cost profile จาก AI client/model ได้ ให้ใช้เป็นสัญญาณอ่อน ๆ เท่านั้น
ไม่ใช่ข้อสรุปเด็ดขาด เช่น ถ้าผู้ใช้เลือกโมเดลประหยัดหรือบอกว่า token แพง ให้ถือ
`C0 Economy` และถามก่อนงานหนัก ถ้าผู้ใช้เลือกโมเดลแรง/ราคาแพงอยู่แล้ว ให้ใช้
`C1` หรือ `C2` ตามความเสี่ยงของงานได้โดยไม่ต้องถามเรื่อง token บ่อย

## Token Cost Estimate Before Heavy Work

ก่อนเริ่มงานที่อาจหนัก ให้ประเมินแบบสั้น:

- `Low`: ตอบจากไฟล์/บริบทไม่กี่จุด, ไม่ต้องรัน test หนัก
- `Medium`: อ่านหลายไฟล์, แก้หลายจุด, มี smoke test
- `High`: scan repo กว้าง, web/research หลายแหล่ง, build/test หลายรอบ, หรือต้องสรุป log/data ยาว

ถ้า cost estimate เป็น `High` และ profile เป็น `C0 Economy` หรือไม่ชัด ให้ถาม
ผู้ใช้ก่อนพร้อมตัวเลือก เช่น:

```text
งานนี้น่าจะใช้ token สูง เลือกได้ 3 แบบ:
1. ประหยัด: อ่านเฉพาะไฟล์หลัก + เสนอแผน
2. มาตรฐาน: แก้และ smoke test เฉพาะจุด
3. ลึก: scan/test ครบเท่าที่จำเป็น
```

ถ้าผู้ใช้สั่งชัดว่า “ทำเลย”, “รันให้เลย”, “เอาให้จบ” ให้ทำต่อได้ แต่ยังต้อง
คุม scope และรายงานว่าใช้โหมดไหนโดยย่อ

## Required Token Behavior

AI ควรเตือนผู้ใช้เมื่อพบว่า:

- โจทย์กว้างจนต้องอ่านเยอะ แต่สามารถเจาะไฟล์/ส่วนที่เกี่ยวข้องได้
- กำลังจะ paste context ซ้ำ ทั้งที่ควรเก็บไว้ใน `.ai/PROJECT_STATE.md`
- กำลังให้ AI อธิบายพื้นฐานที่ผู้ใช้น่าจะรู้แล้ว
- กำลังวนแก้ภาษาโดยไม่มี eval
- กำลังให้ agent scan ทั้ง repo ทั้งที่ task แตะไฟล์ไม่กี่ไฟล์
- log ยาวมาก แต่ต้องการเพียง error 30 บรรทัดสุดท้าย
- งานเริ่มขยายจาก T1/T2 ไปเป็น T3 โดยผู้ใช้ไม่ได้ตั้งใจจ่าย token สำหรับ run ยาว

## Token-Saving Suggestions AI Should Offer

ใช้คำแนะนำสั้น ๆ เช่น:

```text
เพื่อประหยัด token รอบต่อไป ส่งเฉพาะไฟล์ X, error 50 บรรทัดท้าย, และ expected behavior ก็พอ ไม่ต้องส่งทั้ง repo/context เดิม
```

หรือ:

```text
งานนี้ควรอัปเดต .ai/PROJECT_STATE.md หลังจบ เพื่อไม่ต้องเล่า context ใหม่ทุกครั้ง
```

หรือ:

```text
ก่อนให้ Codex รัน deep scan ทั้ง project ลองให้มันอ่าน README + AGENTS + ไฟล์ที่เกี่ยวข้องก่อน จะประหยัดกว่า
```

## Prompt Compaction Protocol

เมื่อจบ session ที่มีสาระ ให้ AI ช่วยอัปเดต `.ai/PROJECT_STATE.md` ด้วยสรุปสั้น ๆ:

- last objective
- last completed work
- current blockers
- next action
- files touched
- commands/tests run
- open decisions
- local resources used

เป้าหมายคือให้ session ถัดไปอ่าน project state แทนการ replay บทสนทนายาว ๆ

## Output Length Control

ถ้าผู้ใช้ไม่ได้ขอรายละเอียด ให้ใช้:

- สรุปก่อน รายละเอียดหลัง
- ไม่อธิบายสิ่งพื้นฐานยาว ๆ
- ใส่ code เฉพาะส่วนที่ต้องเปลี่ยน
- สำหรับ diff ให้สรุปไฟล์และ logic แทน paste ทั้งไฟล์
- ถ้า output ยาวมาก ให้เสนอ structure ก่อนแล้วค่อยลงรายละเอียดเฉพาะส่วนสำคัญ

## Hard Rule

ห้ามลดคุณภาพด้วยการตัดขั้น eval/test/review gate ทิ้งเพื่อประหยัด token

ให้ประหยัดจากการลด context ที่ไม่จำเป็น ไม่ใช่ลดความคิดที่จำเป็น
